# Vector Search — Intuition

Level: Beginner

## One-line intuition

Vector search finds rows that are semantically similar to a query, not just ones that share the same keywords.

## Why this exists

Full-text search finds documents that contain matching words. It cannot find "documents about dogs" when the document says "canines" — no shared words. Vector search converts meaning into numbers (an embedding) and finds the closest numbers — semantic similarity regardless of word choice.

## First-principles explanation

An **embedding** is a list of numbers (a vector) that represents the meaning of text. Embeddings are produced by machine learning models. Texts with similar meaning produce similar vectors (close together in vector space). Texts with different meanings produce different vectors (far apart).

**Vector search** finds the K nearest vectors to a query vector — this is KNN (K-nearest neighbors).

PostgreSQL + pgvector implements this inside the database:

1. Store embeddings in a `vector(n)` column.
2. Use the `<->` operator to compute distance (closer = more similar).
3. Index with `ivfflat` or `hnsw` for fast approximate KNN.

## Micro-concepts

| Concept | Meaning |
|---------|---------|
| Embedding | A vector (list of floats) representing meaning |
| Dimension | Length of the vector (e.g., 768 for BERT, 1536 for OpenAI ada-002) |
| `<->` | L2 (Euclidean) distance — smaller = more similar |
| `<=>` | Cosine distance — smaller = more similar (direction-based) |
| `<#>` | Negative inner product — for dot-product similarity |
| KNN | K-nearest neighbors — find the K most similar vectors |
| `ivfflat` | Approximate index: faster, slightly less accurate |
| `hnsw` | Approximate index: more accurate, higher memory usage |

## Beginner view

Imagine a map where every document is a point:

- "The cat sat on the mat." → point at (0.8, 0.1, 0.3, ...)
- "A kitten rested on a rug." → point at (0.79, 0.12, 0.31, ...) — very close to the first
- "Quantum mechanics equations" → point far away at (0.01, 0.9, 0.7, ...)

A vector search for "cat relaxing on a surface" finds the nearest points — it returns the cat/kitten sentences, not the physics sentence.

```sql
-- pgvector: store and search embeddings
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
    id        SERIAL PRIMARY KEY,
    content   TEXT,
    embedding vector(384)  -- 384-dimensional embedding
);

-- Find the 5 most similar documents to a query embedding
SELECT content, embedding <-> '[0.1, 0.2, 0.3, ...]'::vector AS distance
FROM documents
ORDER BY distance
LIMIT 5;
```

## Intermediate view

**Index types:**

```sql
-- ivfflat: faster to build, slightly less accurate
CREATE INDEX idx_docs_ivfflat ON documents USING ivfflat (embedding vector_l2_ops)
WITH (lists = 100);  -- lists: number of clusters (sqrt of row count is a starting point)

-- hnsw (pgvector 0.5+): more accurate, better recall, higher memory
CREATE INDEX idx_docs_hnsw ON documents USING hnsw (embedding vector_l2_ops)
WITH (m = 16, ef_construction = 64);

-- Set ef_search at query time for recall/speed tradeoff
SET hnsw.ef_search = 100;
```

**Distance operators:**

| Operator | Type | Use when |
|----------|------|---------|
| `<->` | L2 (Euclidean) | General purpose |
| `<=>` | Cosine | Comparing text embeddings (direction matters, not magnitude) |
| `<#>` | Negative inner product | Models trained with dot product similarity |

**Generating embeddings**: pgvector stores vectors but does not generate them. Use:
- An ML model (Ollama locally, OpenAI API, Sentence Transformers)
- Store the result in the `vector` column

## Advanced view

- **Exact KNN** (no index): `ORDER BY embedding <-> query LIMIT k` — returns exact results but is O(n). Fine for small tables (< 100K rows).
- **Approximate KNN** (with index): ivfflat or hnsw — sub-linear search, may miss some true nearest neighbors.
- **Filtering + KNN**: `WHERE category = 'physics' ORDER BY embedding <-> query LIMIT 5` — filtering before KNN is efficient if the index supports it. ivfflat with pre-filtering is less accurate; hnsw handles this better.
- **Hybrid search**: combine FTS ranking + vector distance for best results: `ts_rank(fts, query) * (1 - (embedding <=> query_vec))`.

## Mental model

Think of embeddings as GPS coordinates for meaning. Every piece of text has coordinates in a high-dimensional meaning-space. "Similar meaning" = nearby coordinates. `<->` measures the distance between coordinates. KNN finds the K closest points.

```sql
SELECT content, 1 - (embedding <=> '[0.1, 0.85, 0.72, 0.25]'::vector) AS cosine_sim
FROM documents
ORDER BY embedding <=> '[0.1, 0.85, 0.72, 0.25]'::vector
LIMIT 5;
```

## Non-SQL or hybrid view

Dedicated vector databases (Pinecone, Weaviate, Qdrant, Chroma) specialize in vector search — they offer advanced filtering, multi-tenancy, and GPU-accelerated indexing. The trade-off vs pgvector:

- pgvector: simpler architecture, single database, transactional, good for < 10M vectors.
- Dedicated vector DB: better recall at scale (> 10M vectors), more index options, separate service to manage.

## Design principle

**Start with pgvector; move to a dedicated vector DB only when you hit its limits.** The biggest cost of a separate vector DB is keeping it in sync with the relational database — a consistency problem pgvector avoids by design.

## Critical thinking

- Embedding quality matters more than index tuning. A poor embedding model produces poor search results regardless of index. Validate your embedding model on your specific domain before building a production system.
- The `vector` type requires all vectors in a column to have the same dimension. Changing embedding models (dimension change) requires a full column migration.

## Creative thinking

Store both the content and its embedding in the same row — then you can do hybrid search (FTS + vector) in a single query:

```sql
SELECT
    content,
    ts_rank(fts, text_query) AS text_score,
    1 - (embedding <=> vec_query) AS semantic_score,
    ts_rank(fts, text_query) * 0.3 + (1 - (embedding <=> vec_query)) * 0.7 AS combined_score
FROM documents,
     to_tsquery('english', 'physics') text_query,
     '[0.1, 0.9, 0.7]'::vector vec_query
WHERE fts @@ text_query OR embedding <=> vec_query < 0.3
ORDER BY combined_score DESC;
```

## Systems thinking

In a RAG (Retrieval-Augmented Generation) architecture:

1. User asks a question → generate a query embedding (via Ollama/OpenAI).
2. Vector search → retrieve top-K relevant document chunks from pgvector.
3. Combine chunks + question → send to LLM (Claude, GPT, Llama).
4. LLM generates an answer grounded in the retrieved documents.

pgvector + PostgreSQL handles steps 2 and storage. The local stack (Ollama + Open WebUI) handles embedding generation and the LLM step.

## MCP and agent perspective

An agent using vector search:

1. Receives a user question as text.
2. Calls an embedding tool (Ollama) to convert the question to a vector.
3. Issues the `<->` query to pgvector.
4. Returns the top-K document titles and snippets — not the full embeddings (large, not human-readable).

The MCP tool signature: `search_documents(query: str, limit: int = 5) → list[{id, content, score}]`

The agent role needs only SELECT on the documents table. The embedding is computed in application code — the database only stores and searches vectors.

## Ontology perspective

- An embedding is a **representation** — a learned encoding of meaning as a point in vector space.
- Vector distance is a **metric** on the embedding space — a function that measures similarity.
- KNN is a **nearest neighbor query** — a generalization of range queries to geometric (vector) data.
- `ivfflat` and `hnsw` are **approximate nearest neighbor (ANN) index methods** — they sacrifice exactness for speed.
- pgvector extends PostgreSQL's type system with a new **base type** (`vector`) and **index access methods**.
