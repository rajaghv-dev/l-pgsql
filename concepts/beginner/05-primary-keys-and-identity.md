# Primary Keys and Identity

Level: Beginner

---

## One-line intuition

A primary key is the column (or columns) that uniquely identifies every row in a table — no two rows may have the same value, and no row may leave it empty.

---

## Why this exists

Without a unique identifier, you cannot reliably:
- Point to one specific row from another table
- Update exactly one record
- Deduplicate data
- Build a foreign key relationship

A primary key is the address of a row in the database. Every table should have one.

---

## First-principles explanation

Every fact in a database needs a stable identity. "The book with title 'Dune' by Frank Herbert" is not a reliable identity — two editions could have the same title and author. A surrogate number `id = 42` is: it is assigned once, never changes, and is unique within the table.

A **primary key** is the constraint that enforces this. PostgreSQL automatically:
- Creates a UNIQUE index on the primary key column(s)
- Rejects NULL values in that column
- Rejects duplicate values

---

## Micro-concepts

| Term | Meaning |
|------|---------|
| **Primary key** | The constraint that enforces uniqueness + non-null on a column or set of columns |
| **Natural key** | A key made from real-world attributes (ISBN, email, SSN) |
| **Surrogate key** | An artificial identifier generated by the database (BIGSERIAL, UUID) |
| **Composite key** | A primary key made of two or more columns combined |
| **SERIAL** | Old PostgreSQL shorthand for `INTEGER + sequence + DEFAULT` |
| **BIGSERIAL** | Same but BIGINT — handles up to ~9.2 quintillion rows |
| **GENERATED AS IDENTITY** | SQL-standard way to auto-increment (PostgreSQL 10+) |
| **gen_random_uuid()** | PostgreSQL function that generates a UUID v4 |

---

## Beginner view

Two common approaches:

**Option 1 — Auto-incrementing integer (BIGSERIAL):**
```sql
CREATE TABLE books (
    id    BIGSERIAL PRIMARY KEY,
    title TEXT      NOT NULL
);
-- id is assigned automatically: 1, 2, 3, ...
```

**Option 2 — UUID:**
```sql
CREATE TABLE books (
    id    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    title TEXT NOT NULL
);
-- id is assigned automatically: 'a1b2c3d4-...'
```

For most beginner work: **use BIGSERIAL**. It is simple, readable, and the default in most tutorials.

---

## Intermediate view

### Natural vs surrogate keys

| | Natural key | Surrogate key |
|--|-------------|---------------|
| **Based on** | Real-world attribute (ISBN, email) | Artificial number or UUID |
| **Stable?** | Only if the attribute never changes | Yes — the database never changes it |
| **Meaningful?** | Yes | No (just an address) |
| **Risk** | Natural keys change (email changes, ISBN has editions) | No risk from domain changes |
| **Recommendation** | Use as UNIQUE constraints, not primary key | Use as primary key |

Best practice: always have a surrogate primary key, and separately enforce uniqueness on natural attributes with a UNIQUE constraint.

```sql
CREATE TABLE customers (
    id    BIGSERIAL PRIMARY KEY,       -- surrogate
    email TEXT      NOT NULL UNIQUE    -- natural, enforced as unique
);
```

### SERIAL vs GENERATED AS IDENTITY

`SERIAL` is a PostgreSQL shorthand that creates a sequence and wires it to the column. It works but is non-standard.

`GENERATED AS IDENTITY` is the SQL:2003 standard version, preferred in new schemas:

```sql
id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

Both produce auto-incrementing integers. `GENERATED ALWAYS` prevents manual inserts of the identity column (safer). `GENERATED BY DEFAULT` allows overrides (useful for migrations).

## Advanced view

### UUID trade-offs

| | BIGSERIAL | UUID |
|--|-----------|------|
| **Size** | 8 bytes | 16 bytes |
| **Readability** | `42` | `a1b2c3d4-e5f6-...` |
| **Globally unique** | No | Yes |
| **Index fragmentation** | Low (sequential) | High (random — use UUIDv7 in PG 17+ or `uuid_generate_v1mc`) |
| **Best for** | Single-server apps | Distributed systems, external IDs |

For high-insert-rate tables with UUID PKs, consider `pgcrypto`'s time-ordered UUIDs or UUIDv7 (PostgreSQL 17) to reduce B-tree fragmentation.

### Sequences

`BIGSERIAL` creates a **sequence** object under the hood:
```sql
-- See what sequence backs a table's id
SELECT pg_get_serial_sequence('books', 'id');

-- Peek at the current value
SELECT last_value FROM books_id_seq;

-- Reset a sequence (use with caution)
SELECT setval('books_id_seq', 1000, false);
```

## PostgreSQL view

```sql
-- See the primary key for a table
SELECT constraint_name, column_name
FROM   information_schema.key_column_usage
WHERE  table_name = 'books'
  AND  constraint_name = (
    SELECT constraint_name
    FROM   information_schema.table_constraints
    WHERE  table_name = 'books'
      AND  constraint_type = 'PRIMARY KEY'
  );

-- Or simpler in psql:
\d books
```

## SQL view

```sql
-- BIGSERIAL (traditional, widely compatible)
CREATE TABLE IF NOT EXISTS orders (
    id         BIGSERIAL PRIMARY KEY,
    customer   TEXT      NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- GENERATED ALWAYS AS IDENTITY (SQL standard)
CREATE TABLE IF NOT EXISTS invoices (
    id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    amount     NUMERIC(10,2) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- UUID primary key
CREATE TABLE IF NOT EXISTS sessions (
    id         UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    started_at TIMESTAMPTZ DEFAULT now()
);
```

## Non-SQL or hybrid view

MongoDB uses `_id` (ObjectId — a 12-byte time-ordered unique ID) as its mandatory primary key. DynamoDB requires a partition key and optional sort key. Every database system has a notion of row identity; the implementation differs. PostgreSQL's flexibility (SERIAL, UUID, composite, GENERATED) lets you match the identity strategy to the application.

---

## Design principle

**Default to `BIGSERIAL PRIMARY KEY` unless you have a reason not to.** It is simple, small, sequential (good for index locality), and unambiguous. Switch to UUID when you need global uniqueness across systems or when the ID must be generated client-side before inserting.

---

## Critical thinking

- Can two rows in a table have the same primary key? (No — PostgreSQL enforces uniqueness with an index. An INSERT that violates this raises a unique violation error.)
- What happens to foreign keys that point to a row when that row is deleted? (Depends on ON DELETE behavior — covered in concept 06.)
- Why should the primary key never be updated? (Other tables may reference it via foreign keys. Changing a PK cascades updates or fails, depending on the FK definition. Stability is the whole point of a surrogate key.)

---

## Creative thinking

A primary key is like a government-issued ID number. It is:
- Unique within the issuing authority (the table)
- Never reused after the holder is deleted (gaps in sequences are fine — IDs are not recycled)
- Opaque — tells you nothing about the person (that is a feature, not a bug)

Natural attributes (name, address) change. The ID number does not.

---

## Systems thinking

- **Sequences are per-database** — distributed systems cannot rely on a single sequence for global uniqueness (use UUID)
- **Gaps in BIGSERIAL are normal** — failed transactions consume sequence values; this is by design
- **Primary keys are automatically indexed** — you never need to manually create an index on the PK column
- **Composite PKs** (rare at beginner level) span multiple columns and require all columns to be non-null

---

## MCP and agent perspective

Agents often need to reference rows they created or retrieved earlier. Primary keys are the stable references that make this possible. An agent that stores `customer_id = 42` in its context can later query `SELECT * FROM orders WHERE customer_id = 42` reliably. UUID primary keys are particularly useful for agents that generate IDs client-side before writing to the database.

---

## Ontology perspective

A primary key is an **identifier** in the philosophical sense — it individuates an entity within a class. Two rows with different PKs are different individuals, even if all other attributes are identical. This is the **open-world assumption** boundary: the PK says "this is a distinct thing."
